# Skill Creator

仅在用户明确要求创建或修改 Skill 时使用。先从当前对话提取已经确认的目标、输入、输出和约束；只有关键事实缺失时再简洁询问。不要自动创建、发布或把一次普通回答沉淀为 Skill。

## 设计

1. 用一句话定义 Skill 要解决的重复性任务，以及不适用的情况。
2. 明确触发语句或上下文、输入信息、输出格式、成功标准和失败处理。
3. 选择足够的约束强度：
   - 结果可变、依赖上下文：给出原则和判断条件。
   - 有稳定步骤或参数：给出明确流程和参数。
   - 易出错、需要一致性：给出固定步骤、校验和停止条件。
4. 只记录模型本身不知道、但完成任务确实需要的领域知识或流程。删除背景叙述、系统提示词复述和重复示例。

## 编写

创建的目录必须与 `name` 一致，目录中必须包含 `SKILL.md`。frontmatter 至少包含：

```markdown
---
name: release-review
description: 在用户需要发布前检查变更、验证结果和回滚风险时使用。
---
```

- `name` 只能使用小写字母、数字和连字符；保持简短且表达动作或领域。
- `description` 同时说明能力和触发时机，使用用户可能说出的关键词；不要把完整流程塞进 description。
- 正文使用祈使句，优先写可执行步骤、输入输出和失败边界。
- 保持 `SKILL.md` 精炼。详细参考资料放在 `references/`；确定性或反复使用的程序放在 `scripts/`；产出素材放在 `assets/`。仅在正文中明确说明何时读取或使用它们。
- Skill 没有额外权限。读取文件、执行脚本、联网或写入内容仍必须调用现有 Tool，并遵守审批和安全限制。

## 正文模板

```markdown
# <Skill 标题>

## 目标

<要完成的重复任务与适用边界。>

## 工作流

1. <收集并校验必要输入。>
2. <执行核心步骤。>
3. <验证产出并报告结果。>

## 边界

<缺失输入、风险、失败或需用户确认时的处理。>

## 资源

<仅在需要时引用 scripts/、references/ 或 assets/。>
```

## 草稿与发布

1. 将完成的 `name`、`description` 和正文传给 `create_skill_draft`；不要手写工作区文件。
2. 告知用户草稿已创建并请求审阅；草稿位于项目 `.skills/.drafts/`，不会被 Catalog 使用。
3. 只有用户明确要求发布时才调用 `publish_skill_draft`。
4. 发布前检查名称、目录、frontmatter、正文、资源引用和冲突；发布后让 Runtime 刷新 Catalog。

Turning Good Agent 以项目 `.skills/` 目录作为唯一 Skill 来源，不打包 `.skill` 或 ZIP，也不自动发布。
